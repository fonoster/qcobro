`@qcobro/sdk` is the official TypeScript client for the QCobro API. It wraps the same
tRPC contracts the operator console runs on, so the calls you make from your own backend
are typed end to end and validated before they ever hit the network. Drive QCobro from
your system of record — load portfolios, sync accounts, react to outcomes — without
hand-rolling HTTP.

## How it works

You construct one `Client`, point it at your QCobro endpoint, and call typed resource
methods on it. The client owns the connection and the authentication lifecycle; it talks
tRPC over HTTPS to the QCobro API, which is the very same API the operator console uses.

## The client model

Working with the SDK is always the same four beats: **construct** a client, **authenticate**,
**select a workspace**, then **call** resource methods. This snippet does all four.

```ts
import { Client } from "@qcobro/sdk";

// 1. Construct — the SDK appends the /trpc path to your endpoint itself.
const client = new Client({ endpoint: "https://api.qcobro.com" });

// 2. Authenticate — the issued tokens are stored on the client.
await client.login({ email: "me@acme.com", password: process.env.QCOBRO_PASSWORD! });

// 3. Select the workspace to act in.
client.useWorkspace("ws_abc123");

// 4. Call typed resource methods.
const portfolios = await client.portfolios.list();
```

Almost every method is **workspace-scoped**: the client must be authenticated and have an
active workspace selected before you call it. Selecting a workspace is just
`client.useWorkspace(accessKeyId)` — subsequent calls act within it.

wrapped today; everything else on the API is reachable through the `client.trpc` escape

## Validation happens client-side

Method inputs are validated against the shared `@qcobro/common` schemas **before** any
request is sent. Invalid input — an empty batch, an unknown enum, a malformed email —
throws a structured `ValidationError` with field-level detail and never reaches the
network.

<Note>
  `ValidationError` is re-exported from `@qcobro/sdk`, so you can catch it without
  depending on `@qcobro/common` directly. See [Validation & errors](/sdk/errors) for the
  full error shape.
</Note>

## Tokens and sessions

A single `Client` owns the authentication lifecycle. Authenticate with
`login` (email and password) or `loginWithApiKey` (a workspace API key, for unattended
server-to-server integrations); the issued access and refresh tokens are then stored on
the client and attached to every request.

Tokens are held **in memory only**. By default the client auto-refreshes once on an
`UNAUTHORIZED` response and replays the request, provided it holds a refresh token. To
keep a session alive across restarts, persist the tokens yourself with `client.getTokens()`
and restore them with `client.setTokens(...)`.
